## 主要 PR 順序と重要 design 判断

### 1. #332 gamestate Option B pivot (Phase A + B)

**経緯**: snapshot-per-event (#263) + leak fix (#320 `gc_collect`) + cache (#328 予定) の複雑化を、user の「Option B (`lua.scope` + closure のみ)」で一気に解消提案。#349 plan 前に spike `spike_mlua_scope` で実証、`lua.scope` + `RefCell<&mut World>` の read/write closure 同居 / post-scope error surface / 連続 borrow 衝突なし / double borrow_mut error、すべて期待通り動作確認。

**確定した invariant** (memory `feedback_rust_no_lua_callback.md`):
- `apply::*` (write helper) は `&mut World` のみ、`&Lua` 不使用、Lua value/Function/RegistryKey 触らない、event bus sync fire 禁止
- `build_*_view` (read helper) は `&World` + `&Lua` で table 生成のみ、Lua code 実行禁止
- `try_borrow*` 失敗は `mlua::Error::RuntimeError` 化、panic 禁止
- `fire_event` は queue-only
- Bevy 0.18 の `World::query*` は `&mut World` 要なので、実装上 read closure も `create_function_mut` + `try_borrow_mut` (Lua single-threaded で実害無し)

**Phase B で整理**: 旧 `gamestate_view.rs` 1660 LOC 削除、`run_on_game_start_with_gamestate` / `run_on_game_load_with_gamestate` 新設 (exclusive system)、`_pending_flags` / `_pending_global_mods` drain 撤去、global `set_flag`/`modify_global` 廃止。EffectScope descriptor path は preview 対称性のため保持。

### 2. #334 Command dispatch event-driven (Phase 1-4)

**経緯**: #296 Infrastructure Core 実装で `process_deliverable_commands` が Bevy SystemParam 16-arg cap に到達、dispatcher vs handler 分離の必要性判明 → event-driven refactor 起票 (Option C)。

**Phase 1**: `CommandId` + message 型 + dispatcher + MoveTo handler + CommandLog 2-phase (`Dispatched` → `Executed/Rejected/Deferred`)。Bevy 0.18 の Event→Message 用語変更、`PendingRoute.command_id` を thread、Bevy 20-arm `IntoScheduleConfigs` 上限で `add_systems` 分割。

**Phase 2**: `LoadDeliverable` / `DeployDeliverable` / **CoreDeploy** (`PendingCoreDeploys` resource 廃止、`handle_core_deploy_requested` に deterministic tie-break を migrate) / `TransferToStructure` / `LoadFromScrapyard` / `Colonize` / `Survey` handler。

**Phase 3**: `Scout` handler + `AttackRequested` skeleton (#219/#220 の hook point)、**legacy dispatcher (`process_command_queue` + `process_deliverable_commands`) 完全削除**。

**Phase 4**: Lua bridge `gs:request_command(kind, args)` (ReadWrite only、9 kind 対応、core_deploy/attack は deferred)、`bridge_command_executed_to_gamestate` で `on("macrocosmo:command_completed", fn)` queue-only dispatch、reentrancy safe (`on_command_completed` 内から再 `request_command` 可)。

### 3. #349 ScriptableKnowledge epic (K-1〜K-5)

**設計の core** (memory `project_scriptable_knowledge.md`):
- lifecycle event 自動登録: `define_knowledge { id, payload_schema }` の副作用で `<id>@recorded` / `<id>@observed` を knowledge subscription registry に reserve
- payload mutable on both lifecycles (clamp/sanitize/enrichment、per-observer copy で独立)
- sealed metadata (`kind`/`origin_system`/`recorded_at`/`observed_at`/`observer_empire`/`lag_hexadies`) は `seal_immutable_keys` helper で protect (`__newindex` metatable)
- **同 tick 発火** (plan §0.5 9.1 revised): user の指摘で queue 化しつつも Bevy schedule ordering (`reader.after(writer)`) で同 tick 内 chain 完結
- `HashMap<EventId, Vec<Key>>` (exact) + `HashMap<Lifecycle, Vec<Key>>` (wildcard) の **bucketed registry を v1 から** (O(N) 回避)

**K-5 完結で統合**: 9 `core:*` kind (hostile_detected / combat_outcome / survey_complete / anomaly_discovered / survey_discovery / structure_built / colony_established / colony_failed / ship_arrived) を preload、既存 Rust fact emitter を `PendingKnowledgeRecords` 経由に、`notify_from_knowledge_facts` は test-only として retain (production drain は `dispatch_knowledge_observed` に inline)。

既存 `KnowledgeFact::Hostile` 等 variant は **wire-only 維持** (削除は future cleanup、converter `core_kind_id()` 経由で mechanical 実施可)。

### 4. #326 ESC (ESC-1/2/3)

**framework 設計** (`docs/plan-326-esc.md`):
- `SituationTab` / `OngoingTab` trait + `OngoingTabAdapter<T>` wrapper (blanket impl 回避、coherence-safe)
- `Event` / `Notification` separate type で uniform shape (children tree、leaf vs group 同型)
- `EventKind` closed enum v1、`Custom(String)` 開放は Lua API issue で additive
- `EscNotificationQueue` は既存 `src/notifications.rs` banner queue と **別立て** (統合 scope 外)
- F3 hardcoded、#347 keybinding manager で差替予定

**ESC-2 の notification bridge は最初から Lua wildcard subscriber 形** (当初案の Rust throwaway bridge → K-6 rewrite を廃止、#345 に K-6 scope を統合)。`scripts/notifications/default_bridge.lua` で 9 core:* を `push_notification` に変換。

**ESC-3 4 tab**: Construction Overview (BuildQueue 横断)、Ship Operations (state 分類、戦闘判定)、Diplomatic Standing (`DiplomaticStandingHistory` で前 tick 比較)、Resource Trends (custom sparkline renderer、`ui.painter_at()` line segments、`egui::plot` 依存無し)。

**navigation (click→entity) は未実装** (plan 5 optional、`EventSource::{Ship, BuildOrder, Faction, Empire}` は populate 済、follow-up で handler 追加のみ)。

## Workflow / 運用の気付き

### worktree isolation + plan-first が functional

大 refactor は **Plan agent で plan doc 1 本先行 → impl agent worktree isolation** の 2 段で高成功率。#332 plan-332 (1373 行)、#334 plan-334 (865 行)、#349 plan-349 (1373 行、API 500 途中 error にも関わらず本文は write 済) で実証。各 wave で実装 agent が plan section 参照 + user 確定事項を適用。

### rebase ワークフロー (parallel worktree merge 後)

K-1 ↔ K-3 / ESC-2 ↔ ESC-3 等 parallel PR で `scripting/globals.rs` や `ui/situation_center/mod.rs` の add-only conflict が頻発。先 merge 側に後発 PR を worktree 内で `git rebase origin/main` → conflict 解消 → `git push --force-with-lease`。両方 add-only なので conflict は簡単。

### worktree agent の `cargo fmt` drift 警戒

K-2 worktree 完了後に agent が scope 無しで `cargo fmt` を走らせた形跡あり (17 ファイルに formatting drift が残った)。次 PR 着手前に force-remove で discard、以降 agent prompt で **`cargo fmt` ワークスペース全体禁止、touched file 個別 `rustfmt --edition 2024 <file>` のみ** を強く明示 (memory `feedback_cargo_fmt_scope.md`)。

### Bevy feature prune の効果

`cargo add bevy -F dynamic_linking --package macrocosmo` + `default-features = false` + 16 feature 明示 opt-in で Cargo.lock -973 行、dev build のインクリメンタル時間が大幅短縮。`profile` feature との併存 OK (passthrough は有効時のみ trace_tracy + debug 有効)。

### semantic merge conflict 再発ゼロ

本セッションで parallel PR merge 後の `cargo test --workspace` で regression 0。`feedback_semantic_merge_conflict.md` の規範 (並列 PR は auto-merge できても runtime で conflict 可、統合後 cargo test 必須) が効いている。
