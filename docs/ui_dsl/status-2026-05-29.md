# UI DSL And UI Migration Status - 2026-05-29

This document records the current state of the Lua UI DSL and the ongoing
migration from hand-written Rust egui panels to Lua-defined descriptor
fragments.

## Current Direction

The active direction is no longer a no-op fragment experiment. Lua fragments are
now used by several real UI surfaces, with Rust still owning view extraction,
validation, egui rendering, and command dispatch.

The boundary is:

- Lua owns concrete UI composition, labels, simple layout, grouping, tab lists,
  tooltip content, and command strings.
- Rust owns game-state projection into view tables, descriptor parsing,
  rendering, command validation/dispatch, and fallback behavior.
- `macrocosmo-ui-dsl` must remain independent of Bevy and game-domain types.
- Host crates pass opaque ids and strings into Lua descriptors.

## Crate Shape

`macrocosmo-ui-dsl` currently contains:

- `runtime.rs`
  - `UiFragmentMeta`, source metadata, labels, tags, typed context specs.
  - `UiNode` descriptor tree.
  - Fragment matching/reconciliation/state-update runtime.
  - Layout alignment enums `UiAlignItems` and `UiJustifyContent`.
- `lua.rs`
  - `define_ui_fragment { ... }` accumulator.
  - Lua primitive helper module exposed as `require("macrocosmo.ui")`.
  - Parsing from Lua descriptor tables into `UiNode`.
  - Static tag parsing and context-menu facet indexing.
- `render.rs`
  - Minimal egui renderer for data-only `UiNode` trees.
  - Render stats used by smoke tests.
  - Command click collection, without dispatching inside the DSL crate.

The DSL crate has direct dependency on `egui` and `mlua`; it does not depend on
Bevy, `macrocosmo`, ECS entity types, or simulation modules.

## Lua Surface

The main Lua definitions live in `macrocosmo/scripts/ui/init.lua`.

Current primitives include:

- `section`
- `vstack`
- `hstack`
- `row`
- `grid`
- `text`
- `progress`
- `tabs`
- `tooltip`
- `modified_value`
- `button`
- `action`

`vstack`, `hstack`, and `row` accept:

- `align_items = "start" | "center" | "end"`
- `justify_content = "start" | "center" | "end"`

The current renderer meaning is intentionally conservative. `align_items` is
used for cross-axis placement; `justify_content` is parsed into the descriptor
contract but is not yet fully expressed in egui layout behavior.

Important default: row/hstack alignment defaults to `start`. The top bar opts
into `align_items = "center"` explicitly so the ribbon stays vertically aligned
without making ordinary rows consume excess vertical space.

## Fragment Metadata And Discovery

`define_ui_fragment` supports:

- `id`
- `labels`
- `tags`
- `order`
- `context`
- `render`
- source metadata captured from Lua registration order/source location

Two tag-based discovery paths are currently important:

- `esc_tab = "<tab_id>"`
  - Used by ESC tab hosts to find the fragment intended for each tab.
- `part_of = "context_menu"` plus static context-menu facets
  - Used to precompute context-menu fragment ownership at load time.

The context-menu facet set is intentionally small and load-time validated:

- `part_of = "context_menu"`
- `target = "entity:system" | "entity:ship" | "entity:planet"`
- `ctx:selected = "entity:ship" | "entity:system" | "none"`
- `ctx:selected:ship:class = "colonizer" | "surveyor" | "military" | "transport" | "civilian"`

`command` is allowed as metadata on context-menu fragments, but it is not a
search facet. This keeps query facets limited to values that can be determined
before running dynamic UI logic.

## Migration Status

The following surfaces are wired to Lua fragments with Rust fallback where
appropriate:

- Top bar: `core.ui.top_bar`
- Bottom bar: `core.ui.bottom_bar`
- Notification pills: `core.ui.notification_pills`
- Outline: `core.ui.outline`
- Research window: `core.ui.research`
- Diplomacy window: `core.ui.diplomacy`
- Lua console log body: `core.ui.lua_console`
- Choice dialog body: `core.ui.choice_dialog`
- ESC tabs via `esc_tab` tags
  - notifications
  - construction overview
  - ship operations
  - diplomatic standing
  - resource trends

Context menus are partially migrated:

- Lua now declares command fragments for selected-ship-on-system interactions:
  - `core.ui.context_menu.ship_move_to_system`
  - `core.ui.context_menu.ship_survey_system`
  - `core.ui.context_menu.ship_colonize_system`
- Rust still renders and dispatches the actual context menu.
- The Lua static facet index gates which semantic commands are candidates.
- Dynamic availability checks such as current selection, survey ability,
  colonization ability, and command validity remain in Rust.

The following fragments exist as Lua definitions but are still mostly shadow or
incomplete migration surfaces:

- system panel summary and planet list
- planet window
- colony overview and pop management
- ship multi-selection/detail/refit
- ship designer
- AI debug panels

## UI Issues Recently Addressed

Recent layout issues found during migration:

- Panel headings such as "Top Bar", "Outline", and "Bottom Bar" were removed
  from chrome-like fragments.
- Outline row buttons now use full-width actions.
- Outline system interactions were moved toward the same shared selection and
  context-menu semantics as map interactions.
- Research was restored to tab-style navigation instead of a bare vertical
  panel.
- ESC is a left-side borderless sheet rather than a centered window.
- Disabled button hover can show condition trees.
- Condition tree tooltip rendering now uses explicit tree lines instead of
  ambiguous text indentation.
- Row/hstack alignment is explicit to avoid mixing top-bar vertical centering
  with ordinary content rows.

## Important Limitations

The DSL is usable but still early:

- `justify_content` is parsed but not fully implemented in the renderer.
- There is no general flexbox layout engine. The DSL is still a small egui
  adapter, not a browser layout model.
- Command payloads are still mostly string command ids, not structured typed
  command args.
- Fragment matching for real host slots is still ad hoc. ESC tags and
  context-menu facets are the main concrete discovery paths.
- Context-menu fragments are indexed and used for semantic command discovery,
  but the menu body is not rendered from Lua yet.
- Lua fragments are parsed from the accumulator repeatedly in some draw paths.
  This is acceptable for migration but should become a cached registry/hot
  reload resource.
- Many surfaces still rely on Rust view-table adapters; the DSL crate has no
  game-specific data extraction responsibilities.
- Renderer styling is intentionally minimal and still needs systematic polish.

## Current Validation Commands

These commands were used as the focused validation set around this status:

```sh
cargo fmt
cargo test -p macrocosmo-ui-dsl
cargo test -p macrocosmo --lib ui::context_menu
cargo run -p macrocosmo-ui-dsl --example ui_dsl_render_smoke -- macrocosmo/scripts/ui/init.lua
```

At the time of writing:

- `macrocosmo-ui-dsl` tests pass.
- The targeted `macrocosmo` context-menu compile/test target passes.
- The smoke renderer loads 30 fragments from `macrocosmo/scripts/ui/init.lua`.
- `macrocosmo` still emits many unrelated warnings during targeted tests.

## Next Useful Work

Good next steps:

- Cache parsed Lua UI registries instead of reparsing in each draw path.
- Finish renderer behavior for `justify_content`.
- Render context-menu bodies from Lua fragments once static candidate discovery
  is stable.
- Replace remaining shadow fragments only when the needed view adapter and
  interaction semantics are clear.
- Add a structured action descriptor next to string commands, then migrate
  high-risk actions gradually.
- Expand the catalogue examples to show layout alignment, tabs, condition
  trees, tooltips, modified values, and context-menu facet metadata by purpose.
- Add screenshot or pixel-level smoke coverage for the main migrated panels if
  the project gets a stable headless egui/Bevy harness.
