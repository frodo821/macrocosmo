## 本セッションで確定した重要な設計決定

### (1) 型付き topic bus アーキテクチャ (#195)

`macrocosmo-ai` は **typed topic bus** として実装。metric / command / evidence の 3 トピックに event を emit し、`declare_*` で schema 宣言、`window/current/at_or_before` で query。

- **callback を ai_core に流さない**(一方向依存)
- `subscribe` は「bus がトピック保持する宣言」であり callback 登録ではない
- `log` + `WarningMode::Silent` で運用制御
- 時間逆順 emit / 未宣言 emit / override は `log::warn + no-op(drop)`

記録: `memory/project_ai_core_bus_architecture.md`

### (2) 依存方向反転

v0.2.0 時点: `macrocosmo-ai → macrocosmo`(macrocosmo 型参照)
v0.3.0 決定: **`macrocosmo → macrocosmo-ai`** に反転。ai_core は独自 newtype を持ち、macrocosmo の型を一切参照しない。

整合: CI workflow `ai-core-isolation` が `cargo tree -p macrocosmo-ai` に `bevy` / `macrocosmo` が出現しないことを検証。

### (3) Domain projector パターン (#190 コメント / #191)

AI は単一の巨大アルゴリズムでなく、**ドメイン別 projector が bus に emit、feasibility formula で合成** する。

- 戦闘: Lanchester/Salvo(#190 保留)
- 経済: trajectory projection(#191 完了)
- 外交: PerceivedStanding(#193 完了)
- 技術: 重み付き DAG priority(未実装)
- 領土: influence / pressure map(未実装)

各 projector は pure fn、bus を読んで metric を吐く。ai_core は全体を統合せず、ドメインは疎結合。

### (4) data-first 設計

Objective / IntentKind 等は最初から data struct として設計。enum → data 移行 (#195 issue 原文の Phase 4) は不要化。

### (5) define_deliverable vs define_structure (#223)

- **`define_structure`**: 世界にそのまま存在する構造(debris, wrecks, anomalies、test fixtures)
- **`define_deliverable`**: shipyard 建造 + Cargo 搬送 + Deploy pipeline 経由で置かれる構造

内部表現は共通 `DeliverableDefinition` + `deliverable: Option<DeliverableMetadata>`。shipyard-buildable 判定は `deliverable.is_some()`。

sensor_buoy / ftl_comm_relay / interdictor / defense_platform_kit は `define_deliverable` に移行済。defense_platform(upgrade 出力)は `define_structure`。

### (6) Cargo item 容量 (#223)

`Cargo.items: Vec<CargoItem>` を追加、**既存 `cargo_capacity` を共有プール**として使用。item は `GameBalance.mass_per_item_slot`(Lua 定数、default 1.0 Amt/slot)で Amt 質量に換算。資源と item が同じ mass 空間で競合。

### (7) prerequisites 統一 (#226)

全 definition が `prerequisites: Option<Condition>` に統一:
- `ModuleDefinition.prerequisite_tech: Option<String>` は**完全削除**(後方互換シム無しの hard migration)
- `BuildingDefinition` / `HullDefinition` に追加(従来 silently dropped だった定義が有効化)
- `ShipDesignDefinition` は fieldを持たず `ship_design_effective_prerequisites()` helper で hull + modules から AND 合成

`build_tech_unlock_index` が Building / Hull / Module / ShipDesign 全てを走査、Research パネル表示が整備。

## 運用パターン(今後の並列実行に役立つ)

1. **Plan → Impl 並列**: 複数 issue を Plan agent で同時起動 → 批判的レビュー(open questions 批判)→ Impl agent を worktree isolated で並列起動
2. **worktree agent pollution**: 4-5 割の確率で worktree agent が main に書く bug。pollution monitor(`git status` 30 秒周期ポーリング)で検知 → `git checkout --` + 対応 worktree に apply で rescue
3. **ディスク管理**: merged worktree の `target/` は都度削除(Bevy ビルドは 10-15GB/worktree)。 `git worktree remove` + local/remote branch 削除で完全にクリーンに
4. **merge conflict**: additive な変更(新 struct field、新 enum variant、schema/ids.rs、scripting/mod.rs の `define_*` 登録)は rebase 容易。複数 worktree が同じ `.rs` の違う section に書く場合は trivial 解決
