### 主要パターン

- **Capability-based definitions**: Building / Structure / ShipDesign / Region
- **Lua scope APIs**: ConditionCtx, EffectScope, GameStartCtx, GalaxyGenerateCtx, FtlGraph (record-only, World access なし)
- **KnowledgeStore for remote data**: 全 UI/visualization が遠隔星系は KnowledgeStore 経由、SubLight/Loitering 艦船も光速遅延
- **Async pathfinding**: PendingRoute task + RouteCalculationsPending + FTL blockers (#145 領域)
- **3-phase galaxy generation**: empty → after_phase_a (connectivity, regions) → capitals → init、各段階で Lua フック
- **継続的計画 AI** (#189 設計): long-term objective stable, adaptive per-tick assessment
- **委任パターン** (#189 設計): 中期 AI = NPC 艦隊指揮官 = プレイヤー遠隔地 governor、intent ベース通信
- **非対称 FactionRelations**: (from, to) → FactionView、光速遅延で認識ズレ
- **ROE は 3 層の意味**: Aggressive (追撃)、Defensive (反撃のみ)、Retreat (回避経路)

## v0.3.0 AI 実装の方針 (#189 umbrella 本文、#195 参照)

### クレート分離
- **macrocosmo-ai** 別クレートで実装 (#200 ✅ で雛形済)
- 現状は循環依存 (bin→ai→lib) のため bin から未登録
- #195 で **hook trait** or **macrocosmo-bin 分離** を決定予定

### 3 階層構造 (継続的計画 + 委任)

| 層 | 役割 | state persistence |
|---|---|---|
| **長期計画 (Grand Plan)** | Faction 単位の objective + adaptive assessment | stable objective, adaptive sub-state |
| **中期計画 (Campaign / Governor)** | 艦隊/星系/セクター単位 | intent 受信 + local_objectives + override_log |
| **短期戦術 (Tactics)** | 艦船単位、即応 | state ほぼなし |

### 情報制約
- AI も光速遅延を受ける (KnowledgeStore 経由のみ)
- チートしない (他 faction の真値禁止)
- 「古い情報で立てた計画が新情報到着で再評価される」ドラマが自然発生

### Lua パラメータ
- engine は `ai_params` を generic HashMap<String, LuaValue> で保持
- macrocosmo-ai の `AiParamsExt` trait が具体 key (aggressiveness, objective_persistence, delegation_autonomy 等) を解釈

## 重要な教訓 (本セッション積み残し or 新規)

### worktree agent が main で作業する事象
- 大型 agent の 4-5 割が絶対パスで main の src/ に書いてしまう
- 完了時に `git status` で main を確認、汚染があれば migrate
- ブリーフィングに「相対パス使用 (src/...)」「main 禁止」を明記するが、それでも起きる
- 対策: `setup/` に sentinel を置いて違反時 fail-fast する CI を将来検討

### agent stuck
- 40 分以上 JSONL 無更新 → stuck 判定、`TaskStop` で停止してやり直し
- ブリーフィングに「ブロッカー時は報告して停止 (無限ループ禁止)」を明記して回避率向上

### test output は grep
- `cargo test | tail -20` は中間失敗を見落とす
- 常に `cargo test 2>&1 | grep -E "^test result:|FAILED|^error"` でフィルタ
- 本 pattern を memory に保存済 (feedback_test_output_grep.md)
